# Yutori Scouting Integration — Design

**Date:** 2026-06-06 (revised 2026-06-06 to move package to top-level `apps/scout/` and replace JSON-file storage with DB models)
**Status:** Approved for implementation planning
**Scope:** Yutori → Postgres pipeline only. Email/outreach pipeline is a separate spec.

## Goal

Automatically run a daily research scout against Yutori's Scouting API to discover
new / trending AI benchmark challenges from top-tier conferences, and persist the
results in EvalAI's Postgres database. A future email-outreach subsystem reads
those rows; it is out of scope here.

## Non-goals

- Sending outreach emails (separate spec).
- A web UI for editing the prompt.
- Managing multiple prompts / multiple scouts.
- Implementing webhook delivery retries (Yutori handles retries on its side).

## Background — what the Yutori Scouting API gives us

- `POST https://api.yutori.com/v1/scouting/tasks` with header `X-API-Key: <key>`.
- Body includes `query` (the prompt), `output_interval` (seconds; default 86400),
  `output_schema` (JSON Schema enforced on each run's output), `webhook_url`,
  `webhook_format` (we use `scout`), `skip_email: true`.
- Yutori runs the scout on its own schedule. Results are **not** returned
  synchronously. Each run POSTs results to our `webhook_url`. There is no
  documented polling endpoint.
- The Scouting API does **not** retain memory across runs, so de-duplication
  ("don't repeat previously seen challenges") must happen on our side.

## High-level architecture

```text
   Yutori scout (runs every 24h)
            │
            ▼
   POST <public_base>/api/scout/webhook/<scout-name>/<token>/
            │
            ▼
   scout/views.py: webhook_receiver(request, name, token)
            │
   ┌────────┼─────────────────────────────────────────┐
   │        │                                         │
   ▼        ▼                                         ▼
 lookup   ScoutRun.objects.create(             for each challenge in payload:
 Scout    scout=scout, raw_payload=body)          key = canonical_key(...)
 by name                                          ScoutChallenge.objects.get_or_create(
 + verify                                            canonical_key=key,
 webhook_token                                       defaults={..., source_run=run})
                                                  → `created` flag drives the new-count
```

All persistence is in Postgres via Django models. There is no on-disk JSON storage.

`apps/web/`). Because `apps/` is already on `sys.path` (see
`settings/common.py:20-22`), imports use the bare module name
`scout.X`, matching the existing `accounts`, `web`, `jobs`, etc.


`scout/urls.py`:
```python
from django.conf.urls import url
from . import views

app_name = "scout"
urlpatterns = [
    url(
        r"^webhook/(?P<name>[^/]+)/(?P<token>[^/]+)/$",
        views.webhook_receiver,
        name="webhook",
    ),
]
```

The full webhook URL is
`<SCOUT_PUBLIC_BASE_URL>/api/scout/webhook/<scout-name>/<webhook-token>/`.
Both segments are per-scout: `name` identifies which `Scout` row to look
up, `webhook-token` is the secret stored on that row.

## Settings & secrets

Read from environment (matches EvalAI's existing convention):

| Setting                   | Purpose                                                   |
|---------------------------|-----------------------------------------------------------|
| `YUTORI_API_KEY`          | Sent as `X-API-Key` header to Yutori                      |
| `SCOUT_PUBLIC_BASE_URL`   | Public base URL (e.g. `https://eval.ai` or ngrok URL)     |

The per-scout webhook secret is **not** an env var — it is generated by
`scout_create` (via `secrets.token_urlsafe(32)`) and stored on the
`Scout` row as `webhook_token`. Each scout therefore has its own secret;
compromising one scout's token does not affect any other.

Added to `settings/common.py`. `.env.example` updated with placeholder values.

## Database models (`scout/models.py`)

```python
from base.models import TimeStampedModel
from django.db import models

class Scout(TimeStampedModel):
    """One registered Yutori scout. The schema supports N rows; current
    operational plan keeps a single row named ``"default"``, but
    ``scout_create --name <other>`` registers additional scouts without
    requiring any migration.

    Inherits ``created_at`` and ``modified_at`` from
    ``base.models.TimeStampedModel`` (the EvalAI convention also used by
    ``jobs.Submission`` and other domain models). The inherited
    ``modified_at`` is useful because ``paused_at`` may flip during the
    scout's lifetime.
    """
    name            = models.CharField(max_length=64, unique=True, db_index=True)  # friendly id used in URLs
    webhook_token   = models.CharField(max_length=128)                # per-scout secret (random)
    scout_id        = models.CharField(max_length=64, unique=True)    # UUID from Yutori
    query_hash      = models.CharField(max_length=64)                 # sha256 of prompt at create time
    webhook_url     = models.URLField(max_length=512)
    yutori_view_url = models.URLField(max_length=512, blank=True)
    paused_at       = models.DateTimeField(null=True, blank=True)


class ScoutRun(models.Model):
    """One row per webhook delivery — audit trail of raw Yutori payloads."""
    scout               = models.ForeignKey(
        Scout, on_delete=models.CASCADE, related_name="runs"
    )
    received_at         = models.DateTimeField(auto_now_add=True, db_index=True)
    raw_payload         = models.JSONField()
    new_challenge_count = models.IntegerField(default=0)
    parse_warnings      = models.JSONField(default=list, blank=True)

    class Meta:
        ordering = ["-received_at"]


class ScoutChallenge(models.Model):
    """A deduped benchmark/challenge discovered by the scout."""
    benchmark_name   = models.CharField(max_length=512)
    conference       = models.CharField(max_length=128)
    year             = models.IntegerField()
    canonical_key    = models.CharField(max_length=768, unique=True, db_index=True)

    task_area        = models.CharField(max_length=128, blank=True)
    official_url     = models.URLField(max_length=1024)
    dataset_url      = models.URLField(max_length=1024, blank=True)
    organizers       = models.JSONField(default=list)  # [{name, role, email, affiliation}, ...]
    evalai_suitable  = models.BooleanField()
    evalai_reasoning = models.TextField(blank=True)

    first_seen        = models.DateTimeField(auto_now_add=True, db_index=True)
    outreach_sent_at  = models.DateTimeField(null=True, blank=True, db_index=True)
    source_run        = models.ForeignKey(
        ScoutRun, on_delete=models.SET_NULL, null=True, related_name="challenges"
    )

    class Meta:
        ordering = ["-first_seen"]
```

`ScoutChallenge.outreach_sent_at` is the durable per-row marker the
outreach pipeline (separate spec) uses to decide which organizers still
need an email. It is null on insert and set to `timezone.now()` after
the outreach Celery task sends the emails for that row. There is no
standalone "watermark" table; the absence of a value on the column
itself is the watermark.

## Security model

The webhook URL is `/api/scout/webhook/<name>/<token>/`. The view:

1. Looks up `Scout.objects.get(name=name)`. Not found → **404**, no row
   created, nothing logged beyond a counter.
2. Compares the URL token to the row's stored secret via
   `django.utils.crypto.constant_time_compare(token, scout.webhook_token)`.
   Mismatch → **403**, nothing persisted, nothing logged beyond a counter.
3. Only then does it parse the body and persist a `ScoutRun`.

Each scout therefore has its own secret (a random
`secrets.token_urlsafe(32)` generated by `scout_create`). Compromise of
one scout's token does not affect any other scout. Tokens never appear
in logs.

We do not implement HMAC signature verification because Yutori's
documentation does not describe a signing scheme. If Yutori later
publishes one, swap in HMAC and drop the token-in-path requirement.

CSRF: the view is decorated with `@csrf_exempt` (it's an external webhook).

## Data flow — detail

### One-time setup

```text
$ python manage.py migrate scout
$ python manage.py scout_create                       # registers a scout named "default"
$ python manage.py scout_create --name robotics       # later: register additional scouts
```

`scout_create`:

1. Read `YUTORI_API_KEY` and `SCOUT_PUBLIC_BASE_URL`.
2. Read `--name` argument (default: `"default"`).
3. Refuse to run if a `Scout` row with that exact `name` already exists
   (regardless of paused state). Print a message telling the user to pick
   a different name or delete the existing row. Different `name` values
   are always allowed.
4. Generate `webhook_token = secrets.token_urlsafe(32)`.
5. Build webhook URL: `{SCOUT_PUBLIC_BASE_URL}/api/scout/webhook/{name}/{webhook_token}/`.
6. POST to `https://api.yutori.com/v1/scouting/tasks` with:
   - `query`         = `prompt.RESEARCH_PROMPT`
   - `output_schema` = `schema.OUTPUT_SCHEMA`
   - `output_interval` = 86400
   - `webhook_url`   = built URL
   - `webhook_format` = `"scout"`
   - `skip_email`    = `true`
7. Persist response as a `Scout` row with `name`, `webhook_token`,
   `scout_id`, `query_hash` (sha256 of the prompt at this moment),
   `webhook_url`, `yutori_view_url`.
8. Print `name`, `scout_id`, and `view_url` to stdout.

### Status / pause

```text
$ python manage.py scout_status                          # list ALL scouts + last 5 runs each
$ python manage.py scout_status --name default           # detail for one scout + last 5 runs
$ python manage.py scout_status --name default --pause   # pause via Yutori API; set paused_at
$ python manage.py scout_status --name default --resume  # resume via Yutori API; clear paused_at
```

`--pause` and `--resume` require `--name` so it is always explicit which
scout is being affected.

### Daily webhook delivery

For each POST to `/api/scout/webhook/<name>/<token>/`:

1. **Scout lookup** — `Scout.objects.filter(name=name).first()`. Not found
   → 404, return.
2. **Token check** — `constant_time_compare(token, scout.webhook_token)`.
   Mismatch → 403, return.
3. **Body parse** — JSON decode. Malformed → 400, return.
4. **Open a `transaction.atomic()` block.**
5. **Persist raw** — `run = ScoutRun.objects.create(scout=scout, raw_payload=body)`.
   This happens first so a buggy parser can never lose data.
6. **Extract challenges** — payload is expected to contain
   `{"challenges": [...]}` per our `output_schema`. If the key is missing or
   not a list, log a warning, store it in `run.parse_warnings`, commit the
   transaction, return 200 (Yutori shouldn't retry — the raw row is in DB
   for forensic review).
7. **Merge** — for each challenge dict:
   - Validate it has the schema's required fields. If not, append a warning
     to a `warnings` list and skip.
   - Compute `key = canonical_key(c)`.
   - `obj, created = ScoutChallenge.objects.get_or_create(canonical_key=key, defaults={...all fields..., "source_run": run})`.
   - If `created`, increment a counter.
8. After the loop: set `run.new_challenge_count` and
   `run.parse_warnings = warnings`, save.
9. **Commit** the transaction. Return 200 OK with
   `{"received": N, "new": M, "run_id": run.id, "scout": scout.name}`.

**Idempotency policy:** the view **always** returns 200 if the token is
valid and the body parses, even if some entries are malformed. This
prevents Yutori from retry-storming on partial parse failures. Bad entries
are logged in `run.parse_warnings`, not surfaced as HTTP errors.

**Concurrency:** Postgres handles concurrent inserts. The unique constraint
on `ScoutChallenge.canonical_key` means duplicate webhook deliveries collapse
to no-ops on the second pass (the `get_or_create` returns `created=False`).
The outer `transaction.atomic()` ensures partial failures roll back
cleanly so we never have a `ScoutRun` row with `new_challenge_count` out
of sync with the actual inserted `ScoutChallenge` rows.

Sent as `output_schema` in the scout creation request:

```json
      "type": "array",

        "type": "object",

                     "official_url", "organizers", "evalai_suitable",
                     "evalai_reasoning"],

                             "description": "vision | NLP | multimodal | RL | robustness | medical imaging | speech | other"},

                             "description": "Dataset page or official repo. Empty string if none."},
```

## Dedup key

```python
def canonical_key(c: dict) -> str:
    return f"{c['benchmark_name'].strip().lower()}|{c['conference'].strip().lower()}|{c['year']}"
```

The result is stored in `ScoutChallenge.canonical_key` and protected by a
`unique=True, db_index=True` column constraint. `get_or_create(canonical_key=...)`
is the dedup primitive.

Reasoning: name+conference+year survives URL changes and is robust to
whitespace/case drift. Different challenges at the same venue/year share
the conference + year but differ in name, so collisions are unlikely.

## The research prompt

Lives in `prompt.py` as a module-level constant `RESEARCH_PROMPT`. Verbatim
from the user, with one modification: the "don't repeat the challenge
which you gave me previously" clause is **removed**, because Yutori has no
memory across runs and we enforce dedup locally via the DB unique
constraint. Leaving it in would mislead the model and contaminate every
result.

Changing the prompt = edit `prompt.py`, then:
1. `python manage.py scout_status --name <existing> --pause`
   (pauses the existing scout via Yutori API).
2. `python manage.py scout_create --name <new-name>` (registers a new
   scout under a fresh name with the updated prompt).

No live prompt-editing UI. (When multi-scout-with-per-scout-prompts is
needed, move the prompt to a `Scout.query_text = TextField()` column in
a follow-up migration. That single column addition makes the prompt
DB-editable per scout without changing anything else in this design.)

## Failure modes (handled)

| Scenario                                  | Behavior                                                                |
|-------------------------------------------|-------------------------------------------------------------------------|
| Unknown scout name in webhook URL         | 404, nothing persisted                                                  |
| Bad token in webhook URL                  | 403, nothing persisted                                                  |
| Malformed JSON body                       | 400, nothing persisted                                                  |
| `challenges` key missing/not a list       | 200, `ScoutRun` saved with `parse_warnings`, no `ScoutChallenge` rows added  |
| Single challenge missing required field   | 200, valid entries inserted, bad one recorded in `run.parse_warnings`   |
| Duplicate delivery of identical payload   | 200, second pass yields `created=False` for every challenge; new `ScoutRun` row records `new_challenge_count=0` |
| Same challenge discovered by two scouts   | One `ScoutChallenge` row (canonical_key globally unique); both deliveries get their own `ScoutRun` row, traceable via `run.scout`. The second discovery is a no-op for `ScoutChallenge` (created=False). |
| Yutori API down at `scout_create` time    | Command exits non-zero, no `Scout` row written                          |
| Two webhooks arrive simultaneously        | Postgres handles concurrent inserts; unique constraint enforces dedup   |
| `scout_create` invoked with an existing `--name` | Command aborts with explanatory error (suggesting a different `--name`) |
| Scout deleted while runs exist            | `ScoutRun` rows cascade-deleted (FK has `on_delete=CASCADE`); orphaned `ScoutChallenge` rows survive with `source_run=NULL` (FK has `on_delete=SET_NULL`) |
| Transaction failure mid-loop              | Atomic block rolls back; no partial `ScoutRun` left in DB               |

## Testing strategy

### Unit (`pytest`, no DB)

- `test_dedup.py` — `canonical_key` normalization (whitespace, case, year as int).
- `test_schema.py` — `OUTPUT_SCHEMA` validates against a known-good fixture
  payload using the `jsonschema` library.
- `test_client.py` — `YutoriClient` against a mocked `requests` session:
  correct URL, headers (`X-API-Key`), body shape; surfaces non-2xx as a
  typed exception.

### Webhook view (`pytest-django`, `@pytest.mark.django_db`)

Tests create a `Scout` fixture row before each scenario.

- Valid name + valid token + valid payload → 200, `ScoutRun` row created
  with `scout` FK set, `new_challenge_count` correct, `ScoutChallenge`
  rows inserted, response body has the correct counts.
- Unknown scout name → 404, no DB rows created.
- Valid name + invalid token → 403, no DB rows created.
- Token for one scout sent to another scout's URL → 403, no DB rows
  created. (Confirms tokens are per-scout, not shared.)
- Valid name + token + malformed JSON → 400, no DB rows created.
- Valid name + token + valid payload with one challenge missing a
  required field → 200, valid entries inserted, the bad entry appears in
  `run.parse_warnings`.
- Duplicate delivery (same payload twice to the same scout) → second
  `ScoutRun` has `new_challenge_count=0`; `ScoutChallenge` table count
  unchanged after the second call.
- Same challenge delivered to two different scouts → one
  `ScoutChallenge` row total; both `ScoutRun` rows reference the correct
  scout via `run.scout`.
- Transaction rollback: simulate a DB error mid-loop and confirm no
  partial state remains (the `ScoutRun` row should not exist either).

### Manual end-to-end (before declaring done)

1. `python manage.py migrate scout`.
2. Start ngrok against local Django: `ngrok http 8000`.
3. Set `SCOUT_PUBLIC_BASE_URL` to the ngrok URL.
4. `python manage.py scout_create` against a real Yutori key, with
   `output_interval=1800` (the documented minimum) for the test scout.
5. Wait one cycle; confirm a `ScoutRun` row appears with non-empty
   `raw_payload` and `ScoutChallenge` rows are inserted.
6. `python manage.py scout_status --pause`.
7. Delete the test scout via the Yutori dashboard; bump `output_interval`
   back to 86400 and re-create for production use.
